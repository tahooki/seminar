## Slide 3 - LLM이란?

### 화면 문구

- LLM은 언어를 이해하고 생성하는 모델입니다
- 추론 모델은 생각하는 과정을 거칠 수 있습니다
- 하지만 본질은 글을 작성하는 엔진에 가깝습니다

### 사용 이미지

- `/images/a2ui-poc/generated/slide-02-llm-core.png`

### 스크립트

먼저 LLM부터 짚고 가겠습니다.

LLM은 Large Language Model의 줄임말이고, GPT나 Claude 같은 모델이 여기에 해당합니다.

요즘 추론 모델은 단순히 바로 답을 내는 것이 아니라, 중간에 생각하는 과정을 거쳐서 더 복잡한 문제를 풀 수 있습니다. 그래서 겉으로 보면 사람이 판단하고 계획하는 것처럼 보이기도 합니다.

그런데 본질적으로 LLM은 여전히 **언어를 입력받고, 그다음에 올 가능성이 높은 글을 생성하는 엔진**에 가깝습니다.

즉 LLM은 문장을 이해하고, 정리하고, 추론하고, 설명하는 데 강합니다. 하지만 그 자체만으로 실제 회사 시스템에 들어가서 데이터를 조회하거나, API를 호출하거나, 업무 액션을 실행하는 것은 아닙니다.

그래서 LLM은 Agent의 중요한 두뇌 역할을 하지만, LLM 하나만으로 Agent가 완성되는 것은 아닙니다.

## Slide 4 - LLM만으로 할 수 없는 것들

### 화면 문구

- 사내 DB를 직접 조회하기
- API를 호출해 최신 상태 확인하기
- 파일이나 시스템을 직접 수정하기
- 권한에 따라 실행 여부 판단하기
- 결과를 검증하고 다시 시도하기

### 사용 이미지

- `/images/a2ui-poc/generated/slide-03-llm-limitations.png`

### 스크립트

LLM만으로는 할 수 없는 것들이 있습니다.

예를 들어 LLM은 “지난주 실패한 배포 목록을 알려줘”라는 문장의 의도는 이해할 수 있습니다. 하지만 LLM 혼자서는 실제 배포 DB에 접속해서 최신 데이터를 조회할 수 없습니다.

또 “이 내용을 파일로 정리해줘”라고 했을 때, LLM은 파일에 들어갈 내용을 작성할 수는 있지만 실제 파일 시스템을 읽고 쓰는 것은 별도의 도구가 필요합니다.

API를 호출하거나, 관리자 화면에 등록된 스펙을 확인하거나, 권한이 있는 사용자만 특정 액션을 실행하게 하는 것도 LLM 자체의 역할은 아닙니다.

업무에서는 답을 한 번 만드는 것보다, 결과가 맞는지 확인하고 실패하면 다시 시도하는 과정도 중요합니다.

이런 부분을 해결하려면 LLM 주변에 도구와 실행 구조가 필요합니다.

---

## Slide 5 - 그래서 Agent가 필요합니다

### 화면 문구

- LLM에 도구와 실행 구조를 붙이면 Agent가 됩니다
- 답변 생성에서 업무 수행으로 확장
- 이해 -> 실행 -> 확인 -> 다음 행동

### 사용 이미지

- `/images/a2ui-poc/generated/slide-04-agent-needed.png`

### 스크립트

그래서 Agent라는 구조가 필요합니다.

LLM이 사용자의 요청을 이해하고 판단하는 두뇌라면, Agent는 그 두뇌에 도구와 실행 구조를 붙여서 실제 업무를 진행할 수 있게 만든 시스템입니다.

Agent는 사용자의 목표를 이해하고, 필요한 도구를 고르고, 도구 실행 결과를 다시 확인하고, 다음 행동을 결정합니다.

즉 단순히 “그럴듯한 답변을 작성하는 AI”에서 한 단계 더 나아가, **업무를 진행할 수 있는 AI**로 확장되는 것입니다.

이번 발표에서 말하는 Agent도 이 관점으로 보시면 됩니다.

LLM 자체가 중요한 핵심이지만, LLM에 MCP, Skill, RAG, 상태 관리, 실행 환경, 권한과 검증 같은 요소를 붙여야 실제 서비스에서 쓸 수 있는 Agent가 됩니다.

- Runtime

### 사용 이미지

- `/images/a2ui-poc/generated/slide-05-agent-components-map.png`

### 스크립트

Agent를 구성하는 요소를 한 번에 보면 이렇습니다.

가장 중심에는 LLM이 있습니다. 사용자의 요청을 이해하고, 다음에 무엇을 해야 할지 판단하는 역할입니다.

그다음 MCP와 Tools가 있습니다. Agent가 외부 시스템을 사용할 수 있게 연결해주는 부분입니다.

Skill은 특정 작업을 잘 수행하기 위한 절차와 기준입니다. 단순히 도구만 있는 것이 아니라, 어떤 순서로 보고, 어떤 기준으로 판단하고, 실패하면 어떻게 처리할지를 알려주는 작업법에 가깝습니다.

RAG나 Knowledge는 사내 문서, 정책, 매뉴얼처럼 모델이 기본적으로 알 수 없는 지식을 가져오는 역할입니다.

그 외에도 대화와 작업 상태를 관리하는 State, Agent가 실행되는 Runtime, 권한과 검증을 담당하는 Guardrails가 있습니다.

정리하면 Agent는 LLM 하나가 아니라, LLM을 중심으로 도구, 작업법, 지식, 실행 환경, 권한 검증을 조립한 실행 시스템이라고 볼 수 있습니다.

## Slide 7 - MCP란?

### 화면 문구

- MCP = Agent와 외부 도구를 연결하는 표준 통로
- 어떤 도구가 있는지 알려줍니다
- 도구를 실행하고 결과를 돌려줍니다

### 사용 이미지

- `/images/a2ui-poc/generated/slide-06-mcp.png`

### 스크립트

Agent가 실제 업무를 하려면 외부 시스템과 연결되어야 합니다.

예를 들어 DB를 조회하거나, API를 호출하거나, 파일을 읽거나, Admin에 등록된 스펙을 확인해야 할 수 있습니다.

이때 Agent가 아무 시스템이나 마음대로 호출하는 것이 아니라, 사용할 수 있는 도구 목록과 호출 방식을 정해진 형태로 받아야 합니다.

MCP는 이런 연결을 표준화해주는 통로라고 볼 수 있습니다.

Agent 입장에서는 “내가 어떤 도구를 사용할 수 있는지”, “그 도구에는 어떤 값을 넣어야 하는지”, “실행하면 어떤 결과가 돌아오는지”를 알아야 합니다.

MCP는 이 정보를 Agent에게 제공하고, Agent는 그 도구를 사용해서 실제 업무 데이터를 가져오거나 외부 시스템과 상호작용할 수 있습니다.

## Slide 8 - MCP 사용방법

### 화면 문구

- tools/list
- 도구 목록
- 도구 선택
- tools/call
- 결과 재주입

### 사용 이미지

- `/images/a2ui-poc/generated/slide-07-mcp-usage.png`

### 스크립트

MCP를 실제로 사용할 때는 보통 이런 흐름입니다.

먼저 Agent가 MCP Server에 “내가 사용할 수 있는 도구가 뭐가 있나요?”라고 묻습니다. 이게 `tools/list`입니다.

그러면 MCP Server는 도구 이름, 설명, 입력 스키마를 알려줍니다.

LLM은 사용자 요청과 도구 목록을 보고 어떤 도구를 호출할지 선택합니다.

그다음 Agent Runtime이 `tools/call`로 실제 도구를 실행하고, 실행 결과를 다시 LLM에게 전달합니다.

중요한 점은 LLM이 직접 외부 시스템을 실행하는 게 아니라, LLM은 호출 의도를 만들고 Runtime이 실행한다는 점입니다.

### 사용 이미지

### 스크립트

예를 들어 Figma MCP를 보면 더 이해하기 쉽습니다.

사용자가 “이 Figma 화면을 코드로 구현해줘”라고 요청하면 Agent가 Figma 파일을 그냥 감으로 보는 것이 아닙니다.

즉 MCP는 Agent가 Figma, GitHub, DB 같은 외부 시스템을 사용할 수 있게 해주는 호출 통로입니다.

## Slide 10 - Skill이란?

### 화면 문구

- Skill = 특정 작업을 잘하기 위한 절차와 기준
- 도구를 언제 어떻게 쓸지 알려주는 작업법
- 예시, 템플릿, 검증 방식을 포함할 수 있습니다

### 사용 이미지

- `/images/a2ui-poc/generated/slide-08-skill.png`

### 스크립트

MCP가 도구를 연결해준다면, Skill은 그 도구를 언제 어떻게 써야 하는지 알려주는 작업법에 가깝습니다.

도구만 있다고 해서 Agent가 항상 좋은 판단을 하는 것은 아닙니다.

예를 들어 같은 데이터를 보더라도 전체 목록을 비교해야 하는지, 일부만 훑어보면 되는지, 한 건을 자세히 봐야 하는지에 따라 처리 방식이 달라집니다.

Skill에는 이런 상황별 판단 기준, 작업 순서, 예시, 템플릿, 검증 방식이 들어갈 수 있습니다.

그래서 Skill은 Agent가 매번 처음부터 감으로 판단하지 않고, 특정 작업을 더 일관되고 안정적으로 수행하게 해주는 기준이라고 볼 수 있습니다.

## Slide 11 - Skill 사용방법

### 화면 문구

- SKILL.md
- references/
- templates/
- scripts/
- assets/
- 작업 순서
- 검증 기준

### 사용 이미지

- `/images/a2ui-poc/generated/slide-10-skill-usage.png`

### 스크립트

Skill은 보통 하나의 파일이나 폴더 형태로 관리됩니다.

가장 중심에는 `SKILL.md`가 있습니다. 여기에 이 Skill을 언제 써야 하는지, 어떤 순서로 작업해야 하는지, 주의할 점은 무엇인지가 적힙니다.

필요하면 `references`에는 참고 문서가 들어가고, `templates`에는 결과물 형식이 들어가고, `scripts`에는 반복적으로 실행할 수 있는 보조 스크립트가 들어갑니다.

그래서 Skill은 단순 프롬프트가 아니라, Agent가 특정 업무를 반복해서 잘 수행하도록 만드는 작업 패키지에 가깝습니다.

### 사용 이미지

### 스크립트

다음은 RAG입니다.

RAG는 Retrieval-Augmented Generation의 줄임말입니다.

쉽게 말하면, LLM이 모든 것을 기억한다고 기대하는 것이 아니라, 필요한 문서나 데이터를 먼저 검색해서 가져온 뒤, 그 내용을 LLM에게 context로 넣어 답변하게 만드는 방식입니다.

예를 들어 사내 정책, 운영 매뉴얼, 장애 대응 문서처럼 모델이 기본적으로 알 수 없는 정보는 RAG로 가져와야 합니다.

그래서 RAG는 Agent에게 사내 지식을 연결하는 방법이라고 볼 수 있습니다.

## Slide 16 - A2UI란?

### 화면 문구

- A2UI = Agent-to-User Interface
- Agent가 UI 의도를 전달하고
- Client가 검증된 컴포넌트로 렌더링합니다

### 사용 이미지

- `/images/a2ui-poc/generated/slide-12-a2ui.png`

### 스크립트

A2UI는 Agent-to-User Interface의 줄임말입니다.

핵심은 Agent가 사용자에게 텍스트만 전달하는 것이 아니라, **화면으로 렌더링 가능한 구조화된 UI 정보를 전달한다는 것**입니다.

다만 여기서 중요한 경계가 있습니다.

A2UI는 Agent가 매번 HTML, CSS, JavaScript를 자유롭게 만들어서 실행하는 방식이 아닙니다.

그렇게 하면 보안, 디자인 일관성, 품질 관리가 어려워집니다. 서비스마다 버튼 모양이 달라질 수도 있고, 검증되지 않은 코드가 실행될 수도 있습니다.

이번 POC에서 본 A2UI는 조금 다릅니다.

Agent는 “이 상황에서는 이런 UI가 적합하다”는 의도와 데이터를 전달하고, 실제 화면은 클라이언트가 미리 준비해둔 컴포넌트와 디자인 시스템으로 렌더링합니다.

즉 **Agent는 UI를 선택하고 데이터를 연결하고, Client는 안전하고 일관된 방식으로 화면을 그립니다.**

### 사용 이미지

### 스크립트

이번 A2UI POC를 준비하게 된 배경을 말씀드리겠습니다.

AI Chatbot을 만들면서 가장 많이 했던 고민은 이것이었습니다.

**Chatbot이 사용자에게 더 많은 가치를 주려면 무엇이 필요할까?**

또 앞으로 Agent 기반 개발이 점점 중요해질 것이라고 보고, 팀 차원에서도 Agent 관련 개발 역량을 확보해 나갈 필요가 있다고 판단했습니다.

그래서 Chatbot에 A2UI 기능을 붙여보는 POC를 진행하게 되었습니다.

### 사용 이미지

**Agent에게 UI를 어디까지 맡길 것인가?**

Agent가 모든 화면을 자유롭게 만들게 하면 유연성은 높아집니다. 하지만 결과가 불안정해질 수 있습니다. 디자인도 흔들릴 수 있고, 서비스에서 보장해야 하는 품질을 유지하기 어렵습니다.

반대로 모든 UI를 고정해두면 안정적입니다. 하지만 사용자의 다양한 요청이나 데이터 형태에 유연하게 대응하기 어렵습니다.

그래서 이번 POC에서는 중간 지점을 선택했습니다.

**의미 있는 UI의 외형과 컴포넌트는 미리 만들어두고, Agent는 사용자의 요청과 API 데이터 형식에 따라 가장 적절한 UI를 선택하도록 했습니다.**

이게 이번 POC의 핵심 방향입니다.

## Slide 21 - API 명세와 Data Profile

### 화면 문구

- API Registry
- paramsSchema
- capabilities
- responseHint
- Data Profile
- rowCount / fields / time / metric

### 사용 이미지

- `/images/a2ui-poc/generated/slide-19-api-data-profile.png`

### 스크립트

UI만 등록한다고 끝나는 것은 아닙니다.

Agent가 어떤 데이터를 가져올 수 있는지도 알아야 합니다.

그래서 API 목록을 등록해둡니다. 여기에는 API가 어떤 질문에 답할 수 있는지, 어떤 파라미터를 받는지, pagination이나 filtering이 가능한지 같은 정보가 들어갑니다.

이 부분은 발표에서는 **API 명세**라고 부르겠습니다. 코드 관점에서는 `API Registry`나 `API Contract`에 가깝습니다.

그리고 실제 API 응답이 오면 그 데이터를 바로 UI에 넣는 것이 아니라, 먼저 `Data Profile`을 만듭니다.

Data Profile은 이 데이터가 여러 row인지, 시간 필드가 있는지, 숫자 metric이 있는지, 테이블에 적합한지 차트에 적합한지를 요약한 정보입니다.

### 화면 문구

- 사용자 질문
- API 선택
- 파라미터 추출
- API 실행
- Data Profile
- Render Plan
- SurfaceEnvelope
- SurfaceRenderer

### 사용 이미지

- `/images/a2ui-poc/generated/slide-20-a2ui-runtime-flow.png`

### 스크립트

실제 POC에서는 이런 흐름으로 동작합니다.

사용자가 “지난주 production에서 실패한 배포 목록 보여줘”라고 입력하면, 먼저 이 요청이 Data A2UI로 처리할 수 있는 요청인지 판단합니다.

그다음 API Registry에서 적절한 API를 고릅니다. 이 경우에는 배포 이력 조회 API가 선택됩니다.

이후 사용자의 문장에서 지난주, production, 실패 같은 조건을 파라미터로 추출하고, 등록된 스키마에 맞는지 검증합니다.

검증이 끝나면 API를 실행하고, 응답 데이터로 Data Profile을 만듭니다.

그다음 사용자 의도와 데이터 구조를 보고 Render Plan을 선택합니다.

목록 비교가 필요하면 `table.paginated`, 일부만 훑는 요청이면 `card.simpleList`, 한 건 상세 요청이면 `card.profile`을 선택합니다.

마지막으로 `SurfaceEnvelope`라는 형태로 UI 정보와 데이터를 내려주고, 클라이언트의 `SurfaceRenderer`가 실제 화면을 렌더링합니다.

이렇게 보면 A2UI는 Chatbot 안에서 “읽기 좋은 업무 화면”을 바로 보여주는 역할을 한다고 볼 수 있습니다.

## Slide 25 - POC 화면 3: 크롭 2 - 5개

### 화면 문구

- 크롭 2 - 5개
- “실패한 배포 5건 카드로 보여줘”
- `card.simpleList`

### 사용 이미지

- `/images/a2ui-poc/poc-flow/02-card-list-chat-crop.png`

### 스크립트

다음은 같은 데이터 도메인이지만 요청 방식이 다른 경우입니다.

사용자가 “실패한 배포 5건 카드로 보여줘”라고 요청하면, 전체 목록을 비교하겠다는 것보다 일부 항목을 빠르게 훑어보고 싶다는 의도가 더 강합니다.

그래서 Agent는 같은 API 데이터를 가져오더라도 테이블이 아니라 `card.simpleList`를 선택합니다.

화면에서는 각 배포 건이 카드 단위로 나뉘어 보입니다.

테이블은 여러 항목을 비교하기 좋고, 카드 리스트는 몇 개 항목을 빠르게 훑어보기에 좋습니다.

이 장면에서 보여주고 싶은 포인트는 A2UI가 단순히 “데이터 타입 하나에 UI 하나를 고정 매핑”하는 방식이 아니라는 점입니다.

**같은 API 데이터라도 사용자의 요청 의도에 따라 표현 방식이 달라질 수 있습니다.**

이게 Agent가 UI 선택에 개입하는 지점입니다.

## Slide 26 - POC 화면 4: 크롭 3 - 1개

### 화면 문구

- 크롭 3 - 1개
- “상세 한 개만 보여줘”
- `card.profile`

### 사용 이미지

- `/images/a2ui-poc/poc-flow/03-profile-card-chat-crop.png`

### 스크립트

마지막은 한 건을 자세히 보고 싶은 경우입니다.

사용자가 “실패한 배포 상세 한 개만 보여줘”라고 요청하면, Agent는 여러 row를 비교하는 것보다 한 건의 상세 정보를 정리해서 보여주는 것이 더 적합하다고 판단합니다.

그래서 이번에는 테이블도 아니고 카드 리스트도 아닌, `card.profile` 형태의 단일 상세 카드를 선택합니다.

상세 카드에는 deployment id, service, version, failure stage, failure reason, incident id 같은 정보가 정리됩니다.

이런 정보는 테이블 한 줄 안에 모두 넣으면 읽기 어렵습니다. 반대로 상세 카드로 보여주면 운영자가 “무슨 배포가 왜 실패했는지”를 바로 파악할 수 있습니다.

여기까지 보면 POC의 핵심이 정리됩니다.

**목록이면 테이블, 일부를 훑으면 카드 리스트, 한 건을 깊게 보면 상세 카드.**

즉 A2UI는 AI가 화면을 마음대로 그리는 것이 아니라, **사용자 요청과 데이터 형태를 보고 준비된 UI 중 적절한 것을 선택해 채팅 안에 렌더링하는 방식**입니다.

## Slide 27 - A2UI는 언제 필요할까?

### 화면 문구

- 기존 업무가 복잡할 때
- 기존 업무가 반복적일 때
- 기존 프로젝트를 손보기 어려울 때

### 사용 이미지

- `/images/a2ui-poc/generated/slide-18-a2ui-need.png`

### 스크립트

마지막으로, A2UI는 언제 필요할까요?

A2UI가 모든 화면을 대체해야 하는 것은 아닙니다. 단순한 질문과 답변은 텍스트가 더 빠르고 자연스러울 수 있습니다.

하지만 기존 업무가 복잡한 경우에는 A2UI가 효과적입니다.

예를 들어 사용자가 여러 페이지를 오가면서 데이터를 확인하고, 상태를 비교하고, 다시 다른 화면에서 액션을 해야 하는 업무라면 Chatbot 안에서 필요한 UI를 바로 보여주는 것이 도움이 됩니다.

두 번째는 기존 업무가 반복적일 때입니다.

매번 같은 조건으로 조회하고, 같은 항목을 비교하고, 같은 버튼을 누르는 일이 반복된다면 Agent가 요청을 이해하고 적절한 UI를 바로 보여주는 방식이 업무 시간을 줄일 수 있습니다.

세 번째는 기존 프로젝트를 손보기 어려울 때입니다.

이미 운영 중인 서비스의 화면을 크게 바꾸기는 어렵지만, Chatbot이나 사이드 패널 형태로 A2UI를 붙이면 기존 프로젝트를 크게 수정하지 않고도 플러그인처럼 업무 화면을 확장할 수 있습니다.

그래서 A2UI는 “모든 화면을 AI가 새로 만든다”가 아니라, 기존 업무 흐름 위에 Agent 기반 UI를 덧붙여서 복잡하고 반복적인 일을 줄이는 접근이라고 볼 수 있습니다.

## Slide 28 - 마무리

### 화면 문구

- LLM은 답변을 만들고
- Agent는 일을 진행하고
- A2UI는 결과를 화면으로 보여줍니다

### 사용 이미지

- `/images/a2ui-poc/generated/slide-19-closing.png`

### 스크립트

정리하겠습니다.

첫 번째, LLM은 언어를 이해하고 답변을 생성하는 강력한 엔진입니다. 하지만 실제 업무 데이터를 조회하거나 외부 시스템을 실행하려면 주변 구조가 필요합니다.

두 번째, LLM에 MCP, Skill, RAG, State, Runtime, Guardrails 같은 구성요소를 붙이면 실제 업무를 진행할 수 있는 Agent가 됩니다.

세 번째, A2UI는 Agent의 결과를 텍스트만이 아니라 업무 화면으로 보여주는 접근입니다.

이번 POC에서는 Agent가 화면 코드를 마음대로 생성하는 방식이 아니라, Admin에 등록된 UI와 API 명세를 바탕으로 사용자 요청과 데이터 구조에 맞는 화면을 선택하는 방식을 검증했습니다.

결국 이번 POC의 핵심은 **AI Chatbot을 단순 답변 도구에서 업무 UI Agent로 확장할 수 있는가**였습니다.

이상으로 발표를 마치겠습니다.
