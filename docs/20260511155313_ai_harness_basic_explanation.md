# AI 하네스의 기본 개념

## 핵심 문장

**하네스는 AI에게 원하는 결과를 직접 강요하는 코드가 아니라, AI가 원하는 결과에 가까워질 수밖에 없도록 작업 흐름·제약·검증·재시도 구조를 만들어두는 것이다.**

이 문장을 이해하려면, 하네스를 단순한 `if문`이나 프롬프트 기법으로 보면 부족하다. 하네스는 AI가 일하는 환경 전체를 설계하는 방식에 가깝다.

---

## 하네스가 없는 경우

AI에게 그냥 요청하면 흐름은 단순하다.

```txt
사용자 요청
→ AI 판단
→ 결과 생성
→ 완료
```

이 방식은 빠르지만 결과가 흔들릴 수 있다.

```txt
- 요청을 잘못 해석할 수 있다.
- 중간 설계 없이 바로 결과를 만들 수 있다.
- 결과가 틀렸는데도 완료했다고 말할 수 있다.
- 검증 없이 그럴듯한 답을 낼 수 있다.
- 긴 작업에서는 앞의 목표를 잊을 수 있다.
```

즉, 하네스가 없으면 결과는 대부분 **AI의 즉석 판단**에 의존한다.

---

## 하네스가 있는 경우

하네스를 적용하면 AI의 작업 흐름이 바뀐다.

```txt
사용자 요청
→ 요청 해석
→ 작업 기준 정리
→ 결과 생성
→ 검증
→ 실패 이유 피드백
→ 재시도
→ 완료 기준 통과 후 종료
```

AI에게 자유롭게 맡기는 것이 아니라, AI가 따라가야 할 길을 만들어준다.

```txt
먼저 해석해라.
정해진 형식으로 정리해라.
그 기준에 맞춰 결과를 만들어라.
검증을 통과해라.
실패하면 이유를 보고 다시 고쳐라.
완료 기준을 만족할 때만 끝내라.
```

이게 하네스의 기본 감각이다.

---

## if문과 하네스의 차이

`if문`은 하네스를 구성하는 도구가 될 수 있다. 하지만 `if문` 자체가 하네스는 아니다.

예를 들어 이런 코드는 단순 분기 처리에 가깝다.

```js
if (userWantsCode) {
  generateCode();
} else {
  answerText();
}
```

이건 “어떤 작업을 할지 고르는 코드”다.

하네스에 가까운 구조는 이런 흐름을 가진다.

```js
const spec = await interpret(userRequest);
const result = await generate(spec);
const validation = await validate(result);

if (!validation.ok) {
  return regenerate({
    spec,
    previousResult: result,
    errors: validation.errors,
  });
}

return result;
```

여기서 중요한 것은 `if`가 아니라, **검증 결과를 다시 AI에게 돌려보내는 구조**다.

---

## 프롬프트, if문, 하네스

세 개를 구분하면 이해하기 쉽다.

```txt
프롬프트 = AI에게 말로 부탁하는 것
if문 = 조건에 따라 흐름을 나누는 것
하네스 = AI가 일하는 환경과 절차를 설계하는 것
```

프롬프트는 AI에게 방향을 준다.

```txt
이런 스타일로 만들어줘.
이런 형식으로 답해줘.
이런 점을 조심해줘.
```

if문은 프로그램 흐름을 나눈다.

```txt
요청이 코드 생성이면 코드 생성기로 보낸다.
요청이 요약이면 요약기로 보낸다.
검증이 실패하면 재시도한다.
```

하네스는 이 둘보다 더 큰 구조다.

```txt
요청을 어떻게 해석할지,
중간 결과물을 어떤 형식으로 만들지,
어떤 도구를 사용할지,
무엇을 기준으로 검증할지,
실패하면 어떻게 다시 시킬지,
언제 완료로 볼지
전체 흐름을 설계한다.
```

---

## 하네스의 기본 구성 요소

하네스는 보통 다음 요소들로 구성된다.

```txt
1. 입력 정리
2. 규칙과 제약
3. 중간 산출물
4. 도구 사용
5. 검증
6. 실패 피드백
7. 재시도
8. 완료 기준
```

### 입력 정리

사용자의 요청은 보통 애매하다. 하네스는 이 요청을 AI가 처리하기 좋은 형태로 정리한다.

```txt
사용자 말
→ 의도
→ 조건
→ 필요한 결과물
→ 금지사항
→ 완료 기준
```

### 규칙과 제약

AI가 작업 중 지켜야 할 규칙을 제공한다.

```txt
- 이 형식으로 출력한다.
- 이 도구만 사용한다.
- 이 파일은 수정하지 않는다.
- 이 테스트를 반드시 통과한다.
- 결과에는 이 항목이 반드시 포함되어야 한다.
```

### 중간 산출물

바로 최종 결과를 만들게 하지 않고, 중간 결과를 만들게 한다.

```txt
요청 해석 결과
작업 계획
구조화된 스펙
체크리스트
초안
수정안
```

중간 산출물이 있으면 AI가 무엇을 기준으로 작업했는지 확인할 수 있다.

### 도구 사용

AI가 실제 작업에 필요한 도구를 사용할 수 있게 한다.

```txt
파일 읽기
코드 실행
테스트 실행
검색
데이터 조회
API 호출
결과 렌더링
```

도구는 AI의 말뿐인 판단을 실제 행동으로 바꿔준다.

### 검증

AI가 만든 결과를 외부 기준으로 검사한다.

```txt
- JSON schema에 맞는가?
- 테스트가 통과하는가?
- 타입 에러가 없는가?
- 필수 항목이 있는가?
- 금지된 패턴이 없는가?
- 결과 형식이 맞는가?
```

검증이 있으면 AI가 그럴듯하게 말해도 틀린 결과를 걸러낼 수 있다.

### 실패 피드백

검증에 실패하면 “다시 해”라고만 하지 않는다. 실패 이유를 구체적으로 다시 넣어준다.

```txt
검증 실패:
- 필수 항목 title이 없습니다.
- output 형식이 schema와 다릅니다.
- 테스트 3개 중 1개가 실패했습니다.

수정 지시:
- title 필드를 추가하세요.
- schema에 맞게 output을 다시 작성하세요.
- 실패한 테스트 원인을 수정하세요.
```

### 재시도

한 번에 성공하기를 기대하지 않고, 실패를 감지해서 다시 고치게 한다.

```txt
생성
→ 검증 실패
→ 실패 이유 전달
→ 수정
→ 재검증
→ 통과하면 완료
```

이 반복 구조가 하네스의 핵심이다.

### 완료 기준

AI가 스스로 “완료했다”고 말하는 것을 그대로 믿지 않는다. 완료 기준을 따로 둔다.

```txt
- 테스트 통과
- 필수 파일 생성
- 필수 항목 포함
- 검증 결과 ok
- 변경 사항 요약 완료
```

완료 기준을 통과해야 작업이 끝난다.

## 하네스의 기본 공식

하네스를 가장 단순하게 표현하면 이렇다.

```txt
Harness = Constraint + Feedback + Iteration
```

한국어로 풀면 다음과 같다.

```txt
하네스 = 제약 + 피드백 + 반복
```

AI에게 원하는 방향의 제약을 준다.

```txt
이 형식을 지켜라.
이 도구를 사용해라.
이 기준을 만족해라.
```

결과를 검사해서 피드백을 준다.

```txt
이 부분이 틀렸다.
이 조건을 만족하지 못했다.
이 테스트가 실패했다.
```

다시 시킨다.

```txt
실패 이유를 반영해서 수정해라.
다시 검증해라.
통과할 때까지 반복해라.
```

---

## 하네스는 AI 판단을 없애는 것이 아니다

하네스는 AI의 판단을 전부 코드로 대체하는 것이 아니다.

AI가 잘하는 일은 그대로 맡긴다.

```txt
- 애매한 요청 해석
- 아이디어 생성
- 코드 작성
- 글쓰기
- 오류 원인 추론
- 대안 제안
```

대신 하네스는 AI가 약한 부분을 감싼다.

```txt
- 작업 순서 유지
- 형식 준수
- 금지사항 검사
- 테스트 실행
- 완료 여부 판정
- 실패 후 재시도
```

즉, 하네스는 AI를 억지로 조종하는 장치가 아니라, AI가 더 안정적으로 일하게 만드는 작업 환경이다.

---

## 좋은 하네스의 특징

좋은 하네스는 AI에게 모든 것을 맡기지 않는다.

```txt
AI가 자유롭게 판단할 영역과
반드시 지켜야 할 영역을 분리한다.
```

예를 들어:

```txt
AI가 판단할 것:
- 어떤 표현이 자연스러운가
- 어떤 구조가 더 좋은가
- 어떤 해결책이 적절한가

하네스가 강제할 것:
- 출력 형식
- 필수 조건
- 테스트 통과
- 금지된 행동
- 완료 기준
```

이 분리가 잘 되어야 결과가 좋아진다.

AI에게 모든 것을 강제하면 창의성이 줄어들고, 모든 것을 자유롭게 두면 결과가 흔들린다.

좋은 하네스는 그 사이의 균형을 잡는다.

---

## 한 문장으로 다시 정리

**하네스는 AI에게 “정답을 직접 주입하는 코드”가 아니라, AI가 스스로 생성하고 판단하되, 정해진 절차와 검증을 거치면서 원하는 결과에 가까워지도록 만드는 실행 구조다.**

더 짧게 말하면:

```txt
하네스는 AI를 믿지 않는 구조가 아니라,
AI가 실수해도 다시 고칠 수 있게 만드는 구조다.
```

또는:

```txt
하네스는 프롬프트가 아니라 작업장이다.
```

AI에게 말만 잘하는 것이 프롬프트라면, 하네스는 AI가 일할 수 있는 절차, 도구, 검사 기준, 반복 루프를 갖춘 작업 환경이다.
