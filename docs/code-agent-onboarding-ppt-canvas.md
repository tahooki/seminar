# Code Agent 온보딩 — PPT 캔버스

## 전체 톤
귀여운 AI 로봇이 개발자에게 핵심 개념을 설명하는 크래용 손그림 발표자료.  
딱딱한 기술 연표가 아니라, 개발자가 느낀 불편이 어떻게 다음 기술 흐름으로 이어졌는지를 보여준다.

## 공통 이미지 스타일 프롬프트
```text
귀엽고 친근한 발표용 PPT 슬라이드 한 장면을 만든다.
전체 스타일은 크래용으로 손그림을 그린 듯한 스케치북 느낌이다.
배경은 밝은 크림색 종이 질감이며, 파스텔 블루, 민트, 노랑, 연보라, 연주황을 사용한다.
작은 귀여운 AI 로봇이 발표 도우미처럼 등장해 핵심 내용을 설명한다.
로봇은 둥글고 친근하며, 손짓하거나 말풍선으로 내용을 안내한다.
손그림 화살표, 메모 박스, 밑줄, 동그라미, 귀여운 아이콘을 사용한다.
텍스트는 모두 한글로 표시하고, 너무 많지 않게 유지한다.
발표용 슬라이드답게 핵심 메시지가 한눈에 들어와야 한다.
```

---

# 0번 슬라이드 — 시작 페이지

## 장면설명
발표의 커버 페이지. 귀여운 AI 로봇이 청중을 맞이하며 Code Agent 핵심 개념을 함께 살펴보자고 안내한다. 주변에는 프롬프트, MCP, Skill, AGENTS.md, 하네스 같은 주제 아이콘을 작게 배치한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

시작 페이지처럼 넓은 여백을 가진 커버 슬라이드로 만든다.
가장 큰 제목은 "Code Agent 온보딩".
부제는 "개발자를 위한 핵심 개념 빠르게 이해하기".

중앙에는 귀여운 AI 로봇이 발표를 시작하는 느낌으로 서 있다.
말풍선에는 "함께 시작해봐요!"라고 적는다.

주변에는 작은 손그림 카드로 다음 단어를 배치한다:
"프롬프트", "MCP", "Skill", "AGENTS.md", "하네스".
카드들은 가벼운 점선 화살표로 연결한다.

하단 강조 박스:
"자동완성에서 워크플로 이해까지, Code Agent의 핵심을 한 번에"

전체는 시작 페이지답게 간결하고 귀엽고 여백이 많아야 한다.
```

## 슬라이드 텍스트
- Code Agent 온보딩
- 개발자를 위한 핵심 개념 빠르게 이해하기
- 자동완성에서 워크플로 이해까지

## 발표자 스크립트
오늘은 Code Agent를 단순히 도구 이름 중심으로 보지 않고, 개발자의 작업 방식이 어떻게 바뀌고 있는지 중심으로 보겠습니다. 자동완성에서 시작해 MCP, Skill, AGENTS.md, 하네스까지 이어지는 흐름을 빠르게 잡는 것이 목표입니다.

# 1번 슬라이드 — 개발자는 왜 관심을 가졌나

## 장면설명
개발자가 겪는 반복 작업, 코드 이해, 에러 해결, 도구 이동의 부담을 보여준다. AI 로봇이 “이 불편들이 출발점이에요”라고 설명한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "개발자는 왜 Code Agent에 끌렸을까?"

중앙에는 노트북 앞 개발자와 귀여운 AI 로봇을 배치한다.
로봇은 칠판을 가리키며 문제를 설명한다.

칠판 또는 메모 카드 4개:
"반복 코드"
"낯선 코드베이스"
"에러 추적"
"도구 전환"

하단 강조 문구:
"핵심 관심사는 더 빨리 이해하고, 덜 반복하고, 안전하게 끝내는 것"
```

## 슬라이드 텍스트
- 반복 코드
- 코드베이스 이해
- 에러 추적
- 도구 전환
- 더 빨리, 덜 반복, 더 안전하게

## 발표자 스크립트
Code Agent가 유행한 이유는 개발자의 실제 불편과 맞닿아 있었기 때문입니다. 반복 코드를 줄이고, 낯선 코드를 빨리 이해하고, 에러를 빠르게 추적하고, 여러 도구를 오가는 피로를 줄이고 싶었던 겁니다.

# 2번 슬라이드 — 자동완성: 첫 번째 체감 가치

## 장면설명
AI 로봇이 IDE 옆에서 다음 줄 코드를 제안하는 장면. Copilot류 도구의 가치를 “반복을 줄이는 생산성”으로 설명한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "1단계: 자동완성"

개발자가 IDE에서 코드를 작성하고, AI 로봇이 옆에서 다음 줄을 제안한다.
코드 제안은 말풍선이나 연한 파란색 손그림 박스로 표현한다.

포인트 카드 3개:
"반복 구현 감소"
"초안 작성 속도 향상"
"테스트 코드 진입 장벽 완화"

작은 한계 카드:
"하지만 프로젝트 전체 맥락은 약함"

하단 문구:
"다음 관심사: 코드 전체를 이해해주면 좋겠다"
```

## 슬라이드 텍스트
- 자동완성
- 반복 구현 감소
- 초안 작성 속도 향상
- 전체 맥락은 아직 부족

## 발표자 스크립트
첫 체감 가치는 자동완성이었습니다. 반복 구현과 boilerplate를 줄여주고, 초안 작성 속도를 높여줬습니다. 하지만 개발자는 곧 다음 기대를 갖게 됩니다. 한 줄이 아니라 프로젝트 전체 맥락을 이해해주면 좋겠다는 기대입니다.

---

# 3번 슬라이드 — 코드와 대화하기

## 장면설명
IDE와 채팅창이 함께 있고, 개발자가 코드 구조와 에러를 질문한다. AI 로봇은 관련 파일을 가리키며 설명한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "2단계: 코드와 대화"

IDE 화면, 파일 트리, 채팅 말풍선이 함께 보인다.
AI 로봇은 관련 파일을 손가락으로 가리키며 설명한다.

말풍선 예시:
"이 에러 원인이 뭐야?"
"이 구조를 설명해줘"
"리팩토링 방향은?"

포인트 카드:
"에러 해석"
"구조 이해"
"리팩토링 보조"

하단 문구:
"다음 관심사: 설명 말고 직접 고쳐주면 좋겠다"
```

## 슬라이드 텍스트
- 코드와 대화
- 에러 해석
- 구조 이해
- 리팩토링 보조
- 직접 수정까지 기대

## 발표자 스크립트
다음 단계에서는 코드와 대화하는 경험이 중요해졌습니다. 개발자는 에러 원인을 묻고, 코드 구조를 설명받고, 리팩토링 방향을 상담할 수 있게 됐습니다. 하지만 설명을 받은 뒤 실제 수정은 여전히 사람이 많이 해야 했습니다.

---

# 4번 슬라이드 — 실제 파일을 고치는 Agent

## 장면설명
터미널, git diff, 수정된 파일 카드가 보인다. AI 로봇이 “변경사항을 만들었어요”라고 보여준다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "3단계: 실제 파일 수정"

터미널 창과 git diff 패널을 손그림으로 표현한다.
AI 로봇이 수정된 파일 묶음을 들고 있고, 개발자는 diff를 검토한다.

포인트 카드:
"복붙 감소"
"여러 파일 수정"
"diff 기반 검토"

강조 문구:
"AI 답변을 보는 단계에서, AI 변경사항을 리뷰하는 단계로"

하단 문구:
"다음 관심사: 작은 수정 말고 작업 전체를 맡길 수 있을까?"
```

## 슬라이드 텍스트
- 실제 파일 수정
- 복붙 감소
- 여러 파일 수정
- diff 기반 검토
- 변경사항을 리뷰하는 방식

## 발표자 스크립트
터미널형 Agent가 주목받은 이유는 실제 파일을 고쳐줬기 때문입니다. 이제 개발자는 답변을 복사하는 대신, AI가 만든 diff를 검토합니다. 신뢰의 기준이 답변에서 변경사항으로 옮겨간 것입니다.

---

# 5번 슬라이드 — 작업 루프 전체로 확장

## 장면설명
AI 로봇이 작은 보드에 “계획 → 수정 → 실행 → 테스트 → 복구” 순환 루프를 그려 설명한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "4단계: 작업 루프 전체"

중앙에는 손그림 순환 다이어그램을 크게 배치한다.
단계:
"요구 이해 → 계획 → 수정 → 실행 → 테스트 → 복구"

AI 로봇은 포인터로 루프를 설명한다.
개발자는 옆에서 작업 결과를 확인한다.

포인트 카드:
"긴 작업 처리"
"반복 루프 자동화"
"문제 해결 흐름 지원"

주의 카드:
"권한과 통제가 중요해짐"
```

## 슬라이드 텍스트
- 요구 이해
- 계획
- 수정
- 실행
- 테스트
- 복구
- 권한과 통제

## 발표자 스크립트
이 단계에서 중요한 건 코드 생성이 아니라 작업 루프입니다. 요구를 이해하고, 계획하고, 실행하고, 테스트하고, 실패하면 복구하는 흐름입니다. 이때부터 Agent는 단순 코딩 도구가 아니라 작업을 밀고 가는 파트너처럼 보이기 시작했습니다.

# 6번 슬라이드 — MCP: 도구 연결의 표준화

## 장면설명
AI 로봇이 중앙 허브가 되고, GitHub, 문서, DB, 브라우저, Slack 같은 도구들이 손그림 선으로 연결된다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "5단계: MCP"

중앙에 귀여운 AI 로봇과 큰 카드 "MCP"를 배치한다.
주변 도구 카드:
"GitHub", "문서", "DB", "브라우저", "Slack", "배포"

각 도구는 손그림 점선으로 MCP와 연결한다.

강조 문구:
"AI가 실제 일을 하려면 도구 연결이 필요하다"

포인트 카드:
"연동 방식 표준화"
"사내 시스템 연결"
"확장성 향상"
```

## 슬라이드 텍스트
- MCP
- 도구 연결 표준화
- 사내 시스템 연결
- 확장성 향상
- Code Agent의 연결 포트

## 발표자 스크립트
Agent가 실제 일을 하려면 외부 도구와 연결되어야 합니다. GitHub, 문서, DB, 브라우저, Slack 같은 시스템을 매번 따로 붙이면 복잡해집니다. MCP는 이런 연결 방식을 표준화하려는 흐름으로 볼 수 있습니다.

# 7번 슬라이드 — 실전형 CLI Agent

## 장면설명
개발자와 AI 로봇이 같은 터미널을 보며 테스트를 돌리고 수정한다. 완전 자율보다 “감독하며 함께 일하기”를 강조한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "6단계: 실전형 CLI Agent"

개발자 책상 위에 터미널 창, 테스트 결과, 수정 파일 카드가 보인다.
AI 로봇은 "테스트 다시 돌려볼게요"라는 말풍선을 가진다.

포인트 카드:
"로컬 작업 흐름과 잘 맞음"
"테스트-수정 반복"
"작은 일부터 점진 위임"

강조 문구:
"완전 자율보다, 개발자가 감독하는 Agent"
```

## 슬라이드 텍스트
- CLI Agent
- 로컬 흐름
- 테스트-수정 반복
- 점진 위임
- Supervised Agent

## 발표자 스크립트
실전형 CLI Agent는 개발자의 기존 습관과 잘 맞습니다. 터미널에서 실행하고, 테스트 결과를 보고, 다시 수정하는 흐름이 자연스럽기 때문입니다. 이 단계의 핵심은 완전 자율이 아니라 개발자가 감독하면서 일을 맡기는 방식입니다.

# 8번 슬라이드 — AGENTS.md: 프로젝트 규칙 주입

## 장면설명
AI 로봇이 프로젝트 안내문을 읽고, 테스트 명령어와 코드 스타일을 기억하는 모습.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "7단계: AGENTS.md"

큰 문서 카드에 "AGENTS.md"라고 적는다.
AI 로봇이 문서를 읽고 체크 표시를 한다.

문서 안 항목:
"테스트 명령어"
"수정 금지 폴더"
"코드 스타일"
"PR 전 확인"

강조 문구:
"사람을 위한 README가 있다면, Agent를 위한 안내서도 필요하다"
```

## 슬라이드 텍스트
- AGENTS.md
- 테스트 명령어
- 코드 스타일
- 수정 금지 영역
- PR 전 확인

## 발표자 스크립트
Agent를 계속 쓰다 보면 같은 설명을 반복하게 됩니다. AGENTS.md는 프로젝트 안에서 Agent가 따라야 할 규칙을 담는 안내서입니다. 테스트 명령어, 코드 스타일, 주의사항을 미리 적어두면 반복 실수를 줄일 수 있습니다.

# 9번 슬라이드 — Skills: 반복 작업의 재사용

## 장면설명
AI 로봇이 여러 개의 스킬 카드를 가방에서 꺼내는 장면. “테스트 추가”, “UI 수정”, “로그 분석” 같은 작업 능력을 보여준다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "8단계: Skills"

AI 로봇이 작은 카드 묶음을 펼쳐 보인다.
각 카드는 하나의 작업 능력처럼 보인다.

스킬 카드:
"테스트 추가"
"UI 수정"
"로그 분석"
"배포 체크"

강조 문구:
"반복 프롬프트를 재사용 가능한 작업 방식으로"

비교 메모:
"AGENTS.md = 프로젝트 규칙"
"Skills = 반복 작업 능력"
```

## 슬라이드 텍스트
- Skills
- 테스트 추가
- UI 수정
- 로그 분석
- 배포 체크
- 반복 작업 능력화

## 발표자 스크립트
Skills는 반복 작업을 재사용 가능한 능력으로 만드는 개념입니다. 매번 긴 프롬프트를 쓰는 대신, 테스트 추가나 UI 수정처럼 자주 하는 작업 방식을 Skill로 만들어둘 수 있습니다. 팀의 노하우가 실행 가능한 절차로 바뀌는 셈입니다.

# 10번 슬라이드 — Harness Engineering

## 장면설명
AI 로봇이 작은 모델 코어를 중심으로 문맥, 도구, 권한, 테스트, 평가, 가드레일 카드를 배치한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "9단계: 하네스"

중앙에 작은 카드 "모델"을 놓고, 주변에 보호 장치처럼 여러 카드를 배치한다.
AI 로봇은 설계자처럼 카드들을 정리한다.

주변 카드:
"문맥"
"도구"
"권한"
"테스트"
"평가"
"가드레일"

강조 문구:
"좋은 Code Agent = 좋은 모델 + 좋은 작업 환경"

하단 문구:
"실무에서는 모델보다 운영 구조가 중요해진다"
```

## 슬라이드 텍스트
- 하네스
- 문맥
- 도구
- 권한
- 테스트
- 평가
- 가드레일
- 좋은 모델 + 좋은 작업 환경

## 발표자 스크립트
최근에는 모델 성능만큼이나 주변 환경이 중요해졌습니다. 어떤 문맥을 줄지, 어떤 도구를 허용할지, 테스트와 평가를 어떻게 붙일지가 Agent 품질을 좌우합니다. 하네스는 모델을 실무에서 안전하게 굴리기 위한 작업 환경입니다.

# 11번 슬라이드 — 전체 흐름 정리

## 장면설명
AI 로봇이 긴 길 또는 계단을 따라 Code Agent 발전 단계를 안내한다. 각 단계는 짧은 키워드로만 표시한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "Code Agent 발전 흐름"

왼쪽에서 오른쪽으로 이어지는 손그림 길 또는 계단을 만든다.
AI 로봇이 길을 안내한다.

단계 카드:
"자동완성"
"코드 대화"
"파일 수정"
"작업 루프"
"MCP"
"AGENTS.md"
"Skills"
"하네스"

하단 결론 박스:
"핵심은 코드를 잘 쓰는 AI에서, 개발 환경 안에서 안전하게 일하는 AI로 이동"
```

## 슬라이드 텍스트
- 자동완성
- 코드 대화
- 파일 수정
- 작업 루프
- MCP
- AGENTS.md
- Skills
- 하네스

## 발표자 스크립트
전체 흐름은 자동완성에서 시작해 점점 개발 워크플로 전체로 확장됩니다. 처음에는 코드를 빨리 쓰는 것이 핵심이었지만, 이제는 도구 연결, 프로젝트 규칙, 반복 작업, 운영 구조까지 포함하는 방향으로 발전하고 있습니다.

# 12번 슬라이드 — 마무리 질문

## 장면설명
AI 로봇이 세 가지 질문 카드를 들고 청중에게 던진다. 발표 후 적용 관점으로 마무리한다.

## 장면 프롬프트
```text
[공통 이미지 스타일 프롬프트 포함]

슬라이드 제목: "우리 팀은 어디부터 시작할까?"

AI 로봇이 세 개의 질문 카드를 들고 있다.
배경에는 개발팀, 문서, 터미널, 도구 연결 아이콘이 작게 보인다.

질문 카드 3개:
"반복 작업은 무엇인가?"
"Agent에게 줄 규칙은 무엇인가?"
"안전하게 검토할 루프는 있는가?"

마무리 문구:
"Code Agent 도입은 도구 선택보다, 작업 방식 설계에 가깝다"
```

## 슬라이드 텍스트
- 반복 작업은 무엇인가?
- Agent에게 줄 규칙은 무엇인가?
- 안전하게 검토할 루프는 있는가?
- 도구 선택보다 작업 방식 설계

## 발표자 스크립트
마지막으로 중요한 질문은 어떤 도구를 쓸지가 아닙니다. 우리 팀의 반복 작업은 무엇인지, Agent에게 어떤 규칙을 줄지, 결과를 안전하게 검토하는 루프가 있는지가 더 중요합니다. Code Agent 도입은 결국 작업 방식 설계의 문제입니다.
