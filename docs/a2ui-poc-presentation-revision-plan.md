# Agent와 A2UI 발표자료 수정안

> 목적: 기존 `Agent와 A2UI POC` 발표 흐름을 수정하기 전에, 어떤 내용을 어떻게 바꿀지 먼저 정리한다.
> 범위: 슬라이드 구성, 화면 문구, 발표 스크립트 초안, 이미지 생성 계획, 슬라이드 페이지 반영 계획.

---

## 수정 방향

이번 발표의 앞부분은 바로 Agent를 설명하기보다, 먼저 LLM의 본질과 한계를 설명한 뒤 Agent가 왜 필요한지로 넘어간다.

핵심 흐름은 다음과 같다.

1. LLM은 추론 모델처럼 생각하는 과정을 거칠 수 있지만, 본질적으로는 글을 작성하는 엔진이다.
2. LLM만으로는 실제 업무 시스템을 조회하거나 실행하거나 검증하기 어렵다.
3. 그래서 LLM에 도구, 작업 절차, 상태 관리, 실행 환경, 안전장치를 붙여 Agent를 구성한다.
4. Agent는 이렇게 구성될 때 단순 답변을 넘어 실제 업무를 진행할 수 있다.
5. A2UI는 그 Agent의 결과를 사용자가 이해하고 조작할 수 있는 화면으로 보여주는 방식이다.

## 주요 변경 포인트

### 1. 기존 Agent 초반 설명을 LLM 중심으로 재배치

기존에는 `Agent란?`을 먼저 설명했지만, 비개발자 입장에서는 LLM과 Agent의 차이를 먼저 이해하는 편이 자연스럽다.

따라서 `LLM이란?` → `LLM만으로 할 수 없는 것들` → `그래서 Agent가 필요합니다` 순서로 바꾼다.

### 2. Agent 구성 슬라이드는 전체 지도 역할로 변경

`Agent를 구성하는 것` 슬라이드는 A2UI까지 포함하는 주변 기술 지도가 아니라, Agent 내부 구성요소를 보여주는 슬라이드로 사용한다.

여기서는 다음 요소를 한 화면에 보여준다.

- LLM
- MCP / Tools
- Skill
- State / Memory
- RAG / Knowledge
- Runtime
- Guardrails

이 슬라이드에서는 MCP와 Skill을 자세히 설명하지 않고, 뒤에서 한 페이지씩 더 설명한다고 예고한다.

### 3. MCP와 Skill은 각각 예시 슬라이드를 추가

MCP는 “도구 연결 통로”라는 설명만으로는 추상적일 수 있으므로, 실제 예시 페이지를 추가한다.

Skill도 “작업법”이라는 설명만으로는 감이 약할 수 있으므로, A2UI POC 기준으로 어떤 기준과 절차가 Skill에 들어갈 수 있는지 예시 페이지를 추가한다.

기본 설명 페이지와 예시 페이지의 역할은 분리한다.

- `MCP란?`: MCP의 정의와 역할을 설명한다.
- `MCP 예시`: 실제로 어떤 시스템을 도구로 연결할 수 있는지 보여준다.
- `Skill이란?`: Skill의 정의와 역할을 설명한다.
- `Skill 예시`: A2UI POC에서 Skill이 어떤 판단 기준으로 작동할 수 있는지 보여준다.

### 4. A2UI는 언제 필요할까?는 시연 뒤로 이동

`A2UI는 언제 필요할까?`는 앞쪽 개념 설명 전에 넣기보다, POC 화면 3장을 보여준 뒤 정리하는 위치가 더 자연스럽다.

시연을 먼저 본 뒤, 다음 세 가지 상황에서 A2UI가 특히 필요하다고 정리한다.

- 기존 업무가 복잡할 때: 여러 페이지를 오가면서 작업하는 경우
- 기존 업무가 반복적일 때: 단순 반복 행동을 많이 하는 경우
- 기존 프로젝트를 손보기 어려울 때: 기존 화면을 크게 수정하지 않고 플러그인처럼 확장하고 싶은 경우

## Slide 3 - LLM이란?

### 화면 문구

- LLM은 언어를 이해하고 생성하는 모델입니다
- 추론 모델은 생각하는 과정을 거칠 수 있습니다
- 하지만 본질은 글을 작성하는 엔진에 가깝습니다

### 발표 스크립트

먼저 LLM부터 짚고 가겠습니다.

LLM은 Large Language Model의 줄임말이고, GPT나 Claude 같은 모델이 여기에 해당합니다.

요즘 추론 모델은 단순히 바로 답을 내는 것이 아니라, 중간에 생각하는 과정을 거쳐서 더 복잡한 문제를 풀 수 있습니다. 그래서 겉으로 보면 사람이 판단하고 계획하는 것처럼 보이기도 합니다.

그런데 본질적으로 LLM은 여전히 **언어를 입력받고, 그다음에 올 가능성이 높은 글을 생성하는 엔진**에 가깝습니다.

즉 LLM은 문장을 이해하고, 정리하고, 추론하고, 설명하는 데 강합니다. 하지만 그 자체만으로 실제 회사 시스템에 들어가서 데이터를 조회하거나, API를 호출하거나, 업무 액션을 실행하는 것은 아닙니다.

그래서 LLM은 Agent의 중요한 두뇌 역할을 하지만, LLM 하나만으로 Agent가 완성되는 것은 아닙니다.

---

## Slide 4 - LLM만으로 할 수 없는 것들

### 화면 문구

- 사내 DB를 직접 조회하기
- API를 호출해 최신 상태 확인하기
- 파일이나 시스템을 직접 수정하기
- 권한에 따라 실행 여부 판단하기
- 결과를 검증하고 다시 시도하기

### 발표 스크립트

LLM만으로는 할 수 없는 것들이 있습니다.

예를 들어 LLM은 “지난주 실패한 배포 목록을 알려줘”라는 문장의 의도는 이해할 수 있습니다. 하지만 LLM 혼자서는 실제 배포 DB에 접속해서 최신 데이터를 조회할 수 없습니다.

또 “이 내용을 파일로 정리해줘”라고 했을 때, LLM은 파일에 들어갈 내용을 작성할 수는 있지만 실제 파일 시스템을 읽고 쓰는 것은 별도의 도구가 필요합니다.

API를 호출하거나, 관리자 화면에 등록된 스펙을 확인하거나, 권한이 있는 사용자만 특정 액션을 실행하게 하는 것도 LLM 자체의 역할은 아닙니다.

그리고 업무에서는 답을 한 번 만드는 것보다, 결과가 맞는지 확인하고 실패하면 다시 시도하는 과정도 중요합니다.

이런 부분을 해결하려면 LLM 주변에 도구와 실행 구조가 필요합니다.

---

## Slide 5 - 그래서 Agent가 필요합니다

### 화면 문구

- LLM에 도구와 실행 구조를 붙이면 Agent가 됩니다
- 답변 생성에서 업무 수행으로 확장
- 이해 → 도구 사용 → 결과 확인 → 다음 행동

### 발표 스크립트

그래서 Agent라는 구조가 필요합니다.

LLM이 사용자의 요청을 이해하고 판단하는 두뇌라면, Agent는 그 두뇌에 도구와 실행 구조를 붙여서 실제 업무를 진행할 수 있게 만든 시스템입니다.

Agent는 사용자의 목표를 이해하고, 필요한 도구를 고르고, 도구 실행 결과를 다시 확인하고, 다음 행동을 결정합니다.

즉 단순히 “그럴듯한 답변을 작성하는 AI”에서 한 단계 더 나아가, **업무를 진행할 수 있는 AI**로 확장되는 것입니다.

이번 발표에서 말하는 Agent도 이 관점으로 보시면 됩니다.

LLM 자체가 중요한 핵심이지만, LLM에 MCP, Skill, 상태 관리, 실행 환경, 권한과 검증 같은 요소를 붙여야 실제 서비스에서 쓸 수 있는 Agent가 됩니다.

---

## Slide 6 - Agent를 구성하는 것

### 화면 문구

- LLM
- MCP / Tools
- Skill
- State / Memory
- RAG / Knowledge
- Runtime
- Guardrails

### 발표 스크립트

Agent를 구성하는 요소를 한 번에 보면 이렇습니다.

가장 중심에는 LLM이 있습니다. 사용자의 요청을 이해하고, 다음에 무엇을 해야 할지 판단하는 역할입니다.

그다음 MCP와 Tools가 있습니다. Agent가 외부 시스템을 사용할 수 있게 연결해주는 부분입니다.

Skill은 특정 작업을 잘 수행하기 위한 절차와 기준입니다. 단순히 도구만 있는 것이 아니라, 어떤 순서로 보고, 어떤 기준으로 판단하고, 실패하면 어떻게 처리할지를 알려주는 작업법에 가깝습니다.

그 외에도 대화와 작업 상태를 관리하는 State, 사내 문서나 DB 지식을 가져오는 RAG, Agent가 실행되는 Runtime, 권한과 검증을 담당하는 Guardrails 같은 요소가 있습니다.

오늘은 이 중에서 LLM, MCP, Skill을 먼저 설명드리고, 나머지 요소들은 Agent를 더 안정적으로 만들기 위한 주변 구성으로 이해하시면 됩니다.

---

## Slide 7 - MCP란?

### 화면 문구

- MCP = Agent와 외부 도구를 연결하는 표준 통로
- 어떤 도구가 있는지 알려줍니다
- 도구를 실행하고 결과를 돌려줍니다

### 발표 스크립트

Agent가 실제 업무를 하려면 외부 시스템과 연결되어야 합니다.

예를 들어 DB를 조회하거나, API를 호출하거나, 파일을 읽거나, Admin에 등록된 스펙을 확인해야 할 수 있습니다.

이때 Agent가 아무 시스템이나 마음대로 호출하는 것이 아니라, 사용할 수 있는 도구 목록과 호출 방식을 정해진 형태로 받아야 합니다.

MCP는 이런 연결을 표준화해주는 통로라고 볼 수 있습니다.

Agent 입장에서는 “내가 어떤 도구를 사용할 수 있는지”, “그 도구에는 어떤 값을 넣어야 하는지”, “실행하면 어떤 결과가 돌아오는지”를 알아야 합니다.

MCP는 이 정보를 Agent에게 제공하고, Agent는 그 도구를 사용해서 실제 업무 데이터를 가져오거나 외부 시스템과 상호작용할 수 있습니다.

---

## Slide 8 - MCP 예시

### 화면 문구

- 배포 조회 API 호출
- 사내 DB 검색
- 파일 읽기
- Admin UI 스펙 조회
- 외부 서비스 연동

### 발표 스크립트

MCP를 예시로 보면 더 쉽습니다.

사용자가 “지난주 production에서 실패한 배포 목록 보여줘”라고 말했을 때, LLM은 그 요청의 의미를 이해할 수 있습니다.

하지만 실제 실패한 배포 목록을 가져오려면 배포 조회 API나 DB에 접근해야 합니다.

이때 MCP는 Agent가 사용할 수 있는 도구 목록을 알려주고, 각 도구가 어떤 입력을 받아야 하는지, 실행하면 어떤 결과가 돌아오는지를 표준화된 방식으로 제공합니다.

예를 들어 배포 조회 API, 사내 DB 검색, 파일 읽기, Admin에 등록된 UI 스펙 조회 같은 것들이 MCP를 통해 Agent에게 도구로 제공될 수 있습니다.

즉 MCP는 Agent가 외부 세계와 연결되는 통로라고 볼 수 있습니다.

---

## Slide 9 - Skill이란?

### 화면 문구

- Skill = 특정 작업을 잘하기 위한 절차와 기준
- 도구를 언제 어떻게 쓸지 알려주는 작업법
- 예시, 템플릿, 검증 방식을 포함할 수 있습니다

### 발표 스크립트

MCP가 도구를 연결해준다면, Skill은 그 도구를 언제 어떻게 써야 하는지 알려주는 작업법에 가깝습니다.

도구만 있다고 해서 Agent가 항상 좋은 판단을 하는 것은 아닙니다.

예를 들어 같은 데이터를 보더라도 전체 목록을 비교해야 하는지, 일부만 훑어보면 되는지, 한 건을 자세히 봐야 하는지에 따라 처리 방식이 달라집니다.

Skill에는 이런 상황별 판단 기준, 작업 순서, 예시, 템플릿, 검증 방식이 들어갈 수 있습니다.

그래서 Skill은 Agent가 매번 처음부터 감으로 판단하지 않고, 특정 작업을 더 일관되고 안정적으로 수행하게 해주는 기준이라고 볼 수 있습니다.

---

## Slide 10 - Skill 예시

### 화면 문구

- 어떤 도구를 먼저 사용할지
- 어떤 데이터면 테이블을 고를지
- 어떤 요청이면 카드 UI를 고를지
- 실패했을 때 어떻게 되물을지
- 결과를 어떻게 검증할지

### 발표 스크립트

Skill은 도구 자체라기보다는 작업법에 가깝습니다.

예를 들어 A2UI POC에서 Skill을 만든다면 이런 내용이 들어갈 수 있습니다.

사용자가 목록을 요청하면 먼저 데이터를 조회합니다. 응답이 여러 row이고 비교해야 할 필드가 많으면 테이블 UI를 선택합니다.

사용자가 일부 항목을 빠르게 보고 싶어 하면 카드 리스트를 선택합니다.

한 건의 상세 정보를 요청하면 단일 상세 카드 UI를 선택합니다.

필요한 데이터가 부족하면 바로 실패하지 않고 사용자에게 다시 질문합니다. 그리고 선택한 UI가 요구하는 데이터 스펙과 실제 데이터가 맞는지도 확인합니다.

이런 기준이 있어야 Agent가 매번 감으로 행동하지 않고, 같은 상황에서 비슷한 방식으로 안정적으로 동작할 수 있습니다.

---

## Slide 11 - 그 외 Agent 구성 요소

### 화면 문구

- State / Memory
- RAG / Knowledge
- Runtime
- Guardrails
- Observability

### 발표 스크립트

LLM, MCP, Skill 외에도 Agent를 실제 서비스에서 안정적으로 운영하려면 여러 요소가 필요합니다.

State나 Memory는 지금까지의 대화, 사용자가 이미 말한 조건, 이전 실행 결과를 기억하는 역할을 합니다.

RAG나 Knowledge는 사내 문서, 정책, DB, 매뉴얼 같은 지식을 가져와서 답변이나 판단에 활용하는 방식입니다.

Runtime은 Agent가 실행되는 환경입니다. 모델을 호출하고, 도구를 실행하고, 실패했을 때 재시도하거나 중단하는 흐름을 관리합니다.

Guardrails는 권한, 보안, 검증, 금지된 액션 차단 같은 안전장치입니다.

Observability는 Agent가 어떤 판단으로 어떤 도구를 호출했는지 추적하는 영역입니다.

이런 요소들이 붙을수록 Agent는 단순한 채팅 응답기를 넘어서, 실제 업무에 사용할 수 있는 시스템에 가까워집니다.

---

## Slide 19 - A2UI는 언제 필요할까?

### 화면 문구

- 기존 업무가 복잡할 때
- 기존 업무가 반복적일 때
- 기존 프로젝트를 손보기 어려울 때

### 발표 스크립트

마지막으로, A2UI는 언제 필요할까요?

A2UI가 모든 화면을 대체해야 하는 것은 아닙니다. 단순한 질문과 답변은 텍스트가 더 빠르고 자연스러울 수 있습니다.

하지만 기존 업무가 복잡한 경우에는 A2UI가 효과적입니다.

예를 들어 사용자가 여러 페이지를 오가면서 데이터를 확인하고, 상태를 비교하고, 다시 다른 화면에서 액션을 해야 하는 업무라면 Chatbot 안에서 필요한 UI를 바로 보여주는 것이 도움이 됩니다.

두 번째는 기존 업무가 반복적일 때입니다.

매번 같은 조건으로 조회하고, 같은 항목을 비교하고, 같은 버튼을 누르는 일이 반복된다면 Agent가 요청을 이해하고 적절한 UI를 바로 보여주는 방식이 업무 시간을 줄일 수 있습니다.

세 번째는 기존 프로젝트를 손보기 어려울 때입니다.

이미 운영 중인 서비스의 화면을 크게 바꾸기는 어렵지만, Chatbot이나 사이드 패널 형태로 A2UI를 붙이면 기존 프로젝트를 크게 수정하지 않고도 플러그인처럼 업무 화면을 확장할 수 있습니다.

그래서 A2UI는 “모든 화면을 AI가 새로 만든다”가 아니라, 기존 업무 흐름 위에 Agent 기반 UI를 덧붙여서 복잡하고 반복적인 일을 줄이는 접근이라고 볼 수 있습니다.
